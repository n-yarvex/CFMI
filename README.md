## 摘要

本文聚焦部署于 Cloudflare Workers 边缘节点的复合代理服务，系统解构其核心——VLESS-WS 透明隧道——在协议设计、传输优化、故障恢复及资源管理方面的精妙实现。通过并发竞速、自适应下行整形、上行批量合并及多维健康监测等机制，该隧道在高延迟、不可靠的边缘环境中逼近专用代理的性能极限。同时简要审视 CORS 任意代理的安全代价与实现方式。

---

## 1. VLESS-WS 隧道：从协议到传输的全链路优化

VLESS-WS 隧道利用 WebSocket 帧承载原始 TCP 流，结合 Workers 的原始套接字能力，在客户端与远程主机间构建了一条透明的双向通道。其设计并非简单的数据拷贝，而是深度整合了流量特征感知、动态缓冲管理和确定性的状态转移，形成了一套自适应的传输体系。

### 1.1 协议握手与零-RTT 数据注入

隧道入口首先完成 WebSocket 协议升级，随后立即解析首条二进制消息，消息格式遵循精简的 VLESS 握手规范。握手包含三个关键阶段：

**UUID 身份匹配**：提取 16 字节标识符与预设值进行比对，提供轻量级接入控制。该机制虽不提供加密强度，但足以防范无目的的扫描与滥用，将连接发起权限制在持有 UUID 的客户端。

**多态地址解析**：通过一个类型字段区分 IPv4 字面量、完全限定域名（FQDN）及 IPv6 地址三种格式。解析器根据类型标识读取相应长度的原始字节，动态重构出目标主机字符串和端口号。这种自描述格式无需定界符扫描，解析复杂度恒定为 O(1)，适配边缘环境对延迟敏感的特性。

**首帧载荷直通**：握手数据中携带的原始负载在 TCP 连接建立完成后立即写入套接字，不等待额外的读写回合。这一优化使得交互式会话的首个请求（如 SSH 登录提示）能够节省一个 RTT 的延迟，显著提升用户体验。

### 1.2 并发竞速与中继回退：对抗不可靠性的双重保险

边缘节点与目标主机之间的网络质量波动剧烈，单一的连接尝试很容易被瞬时丢包或路由震荡拖垮。隧道采用“竞速+回退”的复合策略提升连接成功率。

**连接竞速机制**：同时发起多个（可配置，默认 3 个）完全对等的 TCP 连接请求，通过“任一成功即选用”的竞争模式，让最快的路径胜出。其余连接被立即关闭以释放资源。此机制借鉴了 Happy Eyeballs 算法的思想，但不局限于 IPv4/IPv6 选择，而是将并发作为对抗尾部延迟的通用手段，有效压低首包延迟的 P99 指标。

**全球中继回退池**：当所有直连尝试均失败时，系统从预先内置的全球节点列表中随机选取一个中继主机重新发起连接。这些中继节点分布于多个国家和地区，形成一条动态的旁路覆盖网络。随机选取策略避免了单一节点的过载，也增加了路径的不可预测性。这种设计将连接可靠性从单一路径提升至多条独立路径的并联水平。

### 1.3 自适应下行整形：流量感知的三模式缓冲控制

下行方向（TCP→WebSocket）的性能瓶颈在于如何平衡延迟与吞吐。隧道部署了一套基于实时网络特征测量的三模式切换系统。

**特征采样与决策窗口**  
系统以固定周期（1 秒）累计下行传输的字节数和数据包个数，在每个窗口结束时计算平均吞吐量（字节/秒）和平均包大小（字节/包）。这两个指标能够有效区分流量的宏观特征：大文件流通常呈现高吞吐、大包的特征，而远程终端或实时游戏则表现为低吞吐、小包模式。

**三模式定义与切换阈值**  
- **直接模式（Direct）**：当平均吞吐量超过 10 MB/s 且平均包大小超过 16 KB 时激活。此模式下，TCP 读取的数据直接被送入 WebSocket 发送管道，不再经过额外的内存缓冲。零拷贝理念在用户态得以实现，极大减小了内存分配与拷贝开销，适合大带宽文件下载。
- **缓冲模式（Buffered）**：当平均吞吐量低于 2 MB/s 或平均包大小低于 4 KB 时启用。系统积极对下行数据进行合并，类似于 TCP 的 Nagle 算法，但工作在 WebSocket 层。多个小数据包被聚合到接近批处理上限（64 KB）后一次性发送，大幅降低系统调用频率和 WebSocket 帧头开销。此模式非常适合大量小包的交互式流量。
- **自适应模式（Adaptive）**：当网络特征处于两者之间时保持该模式。它根据当前内部缓冲区的饱和度动态决策：若未发送数据超过总缓冲容量的一半，立即执行冲刷；否则启动一个微任务定时器，允许短暂积累后合并发送。该模式在延迟和效率之间寻求瞬时平衡，避免不必要的分组等待。

**振荡抑制设计**  
采样窗口的 1 秒粒度天然形成滞回效应，瞬间的速率抖动不会立即触发模式切换。同时，自适应模式内部的定时器和重试计数机制进一步平滑了缓冲行为，防止在高、低模式间频繁摇摆导致性能退化。

### 1.4 上行批量合并与端到端反压

上行方向（WebSocket→TCP）面临的问题是小而频繁的写入请求可能导致协议栈效率低下。隧道通过一个用户态的合并队列实现写入聚合，并构建了完整的反压传导链。

**合并队列机制**  
队列内部维护一个分片列表，并提供批量弹出（`batch`）操作。当准备写入 TCP 时，队列尽可能将多个连续的小数据块合并到一个预分配的大缓冲区（32 KB）中，然后通过单次 `write()` 调用发出。这种技术相当于在应用层实现了 `TCP_CORK` 的效果，有效减少细碎写入引起的协议栈处理开销和网络上的小包泛滥。

**反压传播链路**  
合并队列设定了总字节上限（128 KB）和最大分片数量。当到达任一限制时，新数据的入队操作会立即失败，并向上层返回反压信号。隧道将这一信号解释为连接不可恢复的拥塞，触发整体连接关闭，从而将压力传导至客户端。这一设计避免了在慢速后端情况下无限制地堆积数据，保证了 Worker 内存使用的有界性，实现了端到端的流控制闭环。

### 1.5 多维健康监测与自愈机制

长连接在中间设备林立的公网环境中极易因空闲超时、路径切换等原因中断。隧道部署了独立的读写健康监测器，配合自动重连实现故障自愈。

**应用层心跳保活**  
以 5 秒为周期检查最后写活动时间，若空闲超过 15 秒，则向 TCP 套接字写入一个零长度数据。此空写操作不传递应用数据，但会刷新沿途 NAT 和状态防火墙的连接跟踪表，有效阻止因空闲而被断开。心跳周期与空闲阈值的不对等设计避免了频繁的空包发送。

**停滞检测与累积判定**  
独立的读监测器以 4 秒为周期计算距离最后成功读取的时间。若超过 8 秒无数据流入，累计一次停滞事件；当连续停滞计数达到 12 次（即总计约 96 秒无数据）时，判定连接已死。这种“宽松超时+连续计数”的设计能够容忍服务端的短暂静默（如无输出的空闲终端），却可以可靠地捕捉真正的路径中断，避免了因误判而频繁重启连接。

**重连与队列持久化**  
连接断开后，系统立即清理旧句柄并启动异步重连流程。在重连期间到达的上行数据仍被暂存在合并队列中。一旦新连接建立成功，队列中的数据会被优先批量发送，保证了断线时刻数据的完整性。重连次数上限（15 次）避免无限循环消耗 CPU 和内存，超过上限后则执行彻底的资源释放。

### 1.6 确定性资源生命周期管理

由于 Workers 环境不支持优雅的析构机制，隧道通过统一的拆卸函数（`teardown`）在所有退出路径上执行清理操作。清理顺序严格定义为：停止定时器、清空缓冲区、释放写入锁、关闭 TCP 套接字、关闭 WebSocket。每个步骤均用异常捕获隔离，确保单点失败不阻塞后续资源的释放，实现了确定性的资源回收，杜绝内存泄漏或句柄残留。

## 2. CORS 任意代理：极端开放设计的双刃剑

CORS 代理通过查询参数接受目标 URL，将请求转发并将响应注入 CORS 头后返回，使任意网页可以突破同源策略限制。其实现包含几项关键设计：

- **安全头剥离**：自动删除 13 种安全相关响应头（如 CSP、X-Frame-Options），使得被代理页面可被嵌入 iframe 或被任意脚本访问。
- **协议升级代理**：检测到 WebSocket 升级请求时，通过内置的 WebSocketPair 建立双向中继，将客户端与上游 WebSocket 服务透明桥接。
- **重定向保持**：对 3xx 响应中的 Location 头进行重写，确保后续请求继续经由代理，维持整个会话的代理上下文。

该代理将便捷性置于首位，但完全移除了目标站点的安全屏障，仅在高度受控或明确授权的场景下适用。

## 3. 总结

本服务在 Cloudflare Workers 的受限容器内，通过一系列协同工作的优化策略，构建了一个既适合高吞吐下载又兼顾低延迟交互的自适应隧道。并发竞速和回退中继保证了连接的高可用性，自适应下行整形和上行合并队列实现了网络层面的高效利用，多维健康监测与自愈机制则赋予了长时间稳定运行的能力。
