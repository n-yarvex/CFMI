## 一、VLESS-WS 隧道实现亮点

### 1. 协议握手与地址解析

隧道入口首先通过 WebSocket 升级建立连接，随后解析客户端发来的首条二进制消息。消息格式严格遵循 VLESS 协议：前 16 个字节为 UUID，用于身份匹配；紧接着是地址长度、端口和地址类型字段。地址类型支持 IPv4、域名和 IPv6 三种，可灵活适应不同的后端目标。解析成功后，剩余的数据将作为初始载荷立即推送到 TCP 连接，实现握手即传数据，消除了额外的往返延迟。

UUID 验证是整个隧道最基础的访问控制手段。只有提供了匹配 UUID 的客户端才能建立转发，虽然它不是强加密认证，但足以阻止未授权的随意连接。

### 2. 连接建立的双重保障：竞速与回退

当隧道需要与目标主机建立 TCP 连接时，系统会同时发起多个并发的连接尝试，并只接受第一个成功的连接。这种“连接竞速”机制可以有效规避网络中单条路径的延迟波动或丢包，显著缩短首包响应时间。

如果所有直连尝试都失败，系统会立即从预先配置的全球代理节点池中随机选取一个中继节点进行二次尝试。这一回退策略保证了即使目标地址不可达，流量也能通过中继节点绕行，避免隧道完全中断。代理池包含分布在多个国家和地区的节点，提供了充足的冗余路径。

### 3. 自适应下行缓冲策略

这是本隧道最核心的性能优化设计。下行方向的数据从 TCP 流向 WebSocket 时，系统会根据实时网络状态动态选择三种传输模式：

- **直接模式**：当检测到带宽充足且数据包较大时，数据会绕过缓冲区直接发送，追求极致的吞吐量和低延迟。
- **缓冲模式**：当带宽较低或数据包普遍较小时，系统会积极合并小包，攒满一定量再一次性发送，以减少协议开销和系统调用次数。
- **自适应模式**：在一般网络条件下，系统会根据当前缓冲区的饱和度智能决策。如果积累的数据已经超过缓冲区一半，就立即发送；否则会短暂等待更多数据到达后再合并发送，从而在延迟和效率之间取得平衡。

模式切换由一套实时采样机制驱动。系统每秒计算一次下行数据的平均速度和平均包大小，并与预设的阈值进行比较。当速度超过每秒十兆字节且平均包大小超过十六千字节时，自动升档到直接模式；当速度低于每秒两兆字节或平均包大小低于四千字节时，降档到缓冲模式。这种反馈控制能确保隧道在各种网络条件下都保持最佳性能，避免了固定缓冲策略可能带来的队头阻塞或过度延迟。

### 4. 上行数据合并队列

上行方向（客户端到 TCP）同样采用了批量合并技术。系统维护一个智能队列，可以将多个小块数据合并成一个不超过三十二千字节的大块，再一次性写入 TCP 连接。这样大幅减少了网络写入的系统调用次数，提高了上行吞吐量。

队列还具备反压控制能力。当未发送的数据总量超过一百二十八千字节时，新到达的数据会被拒绝接收，进而触发上层流控或断连，防止在慢速后端的情况下内存无限增长。合并操作使用了预分配的缓冲区和零拷贝视图，进一步减少了内存分配开销。

### 5. 心跳与停转检测

为了保持长连接活跃并及时发现僵死连接，系统启用了两个独立的定时器：

- **心跳定时器**：每隔五秒检查一次，如果距离上次成功写入超过十五秒，就会发送一个长度为空的保活消息。这能有效防止中间网络设备因长时间无流量而关闭连接。
- **停转检测定时器**：每隔四秒检查一次读取活动。如果超过八秒没有收到任何数据，就计为一次停转。连续停转计数达到十二次后，系统判定连接已经僵死，立即启动重连流程。

这种双重阈值与累积次数的设计，避免了单次网络抖动就被误判为断线，保证了检测的准确性和及时性。

### 6. 自动重连与数据无丢失

当 TCP 连接意外断开（无论是因网络故障、后端关闭还是检测到僵死），系统会自动进入重连过程。重连时，系统会重新发起对同一目标（或回退节点）的连接，并将暂存在上行队列中的数据重新发送，确保在断线期间到达的客户端数据不会丢失。

重连机制受次数限制，最多尝试十五次，超过后才会彻底关闭隧道，防止无限重试浪费资源。整个重连过程是一个异步状态机，由统一的任务调度函数驱动，能够安全处理并发事件，避免重复连接或状态混乱。

### 7. 完备的资源清理

无论隧道以何种方式终止（正常关闭、错误、超时），系统都确保所有资源被安全释放。清理流程会逐一尝试停止所有定时器、释放底层 TCP socket 和 WebSocket 的锁与连接，并使用异常捕获保护每个步骤，防止某个清理操作失败导致整体泄露。

## 二、CORS 任意代理功能

CORS 代理是一个独立的模块，通过查询参数接收目标 URL，发起 HTTP 或 HTTPS 请求并将结果返回给浏览器，同时强制添加允许任意来源的 CORS 响应头，从而使网页能够跨域请求任何资源。

代理在转发时，会生成丰富的浏览器请求头，降低被目标站点识别的风险。它还特意删除了目标服务器设置的十三种安全响应头，例如内容安全策略、框架选项、严格传输安全等，从而允许将目标页面嵌入 iframe 或进行更自由的跨站操作。对于重定向响应，代理会自动改写 Location 头，确保后续请求仍然经过代理，不会脱离代理上下文。Cookie 的处理也遵循标准方法，能够完整保留所有属性。此外，代理还支持 WebSocket 协议的透明中继，通过连接配对实现客户端与上游 WebSocket 之间的双向数据转发。

该代理被设计为“最开放”模式，旨在最大限度地解除浏览器的同源限制，但这也带来了显著的安全风险，必须谨慎使用。
